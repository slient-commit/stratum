Stratum Documentation

A modular core framework for building domain-specific applications with Node.js and React. Everything is a module. Everything can be swapped.

Fully Modular

Database, auth, API, UI — every capability is a self-contained module that can be enabled or disabled in one config file.

Service-Oriented

Modules communicate through a shared service registry and event bus. No hard-wired imports between modules.

Lifecycle-Driven

Modules go through ordered phases — register, init, start, stop, destroy — with automatic dependency resolution.

Swap-Ready

Replace SQLite with Firestore, JWT auth with Firebase Auth — downstream modules work without code changes.

Quick Start

1. Install

npm install

2. Start Backend

node index.js

This boots the Kernel, loads all enabled modules, runs migrations, and starts Express on port 3001.

3. Start Frontend

npm run dev:ui

Vite dev server on port 3000, proxied to the backend. Open http://localhost:3000.

4. Test the API

# Register
curl -X POST http://localhost:3001/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","email":"admin@test.com","password":"secret123"}'

# Login
curl -X POST http://localhost:3001/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"secret123"}'

# Get current user (use the token from login)
curl http://localhost:3001/api/auth/me \
  -H "Authorization: Bearer <token>"

Kernel

The Kernel is the central orchestrator. It wires together the ConfigManager, ModuleLoader, EventBus, ServiceRegistry, and LifecycleManager.

Boot Sequence

stratum.config.js │ ▼ ┌─────────────┐ ┌──────────────┐ ┌────────────────────┐ │ ConfigManager│────▶│ ModuleLoader │────▶│ LifecycleManager │ │ (parse config)│ │ (require, │ │ (run phases with │ │ │ │ validate, │ │ context for each │ │ │ │ topo-sort) │ │ module) │ └─────────────┘ └──────────────┘ └────────────────────┘ │ ┌─────────────┼──────────────┐ ▼ ▼ ▼ register() init() start() │ │ │ ▼ ▼ ▼ declare services connect DBs mount routes set up events run migrations go live
index.js
const { Kernel } = require('@stratum/core');
const config = require('./stratum.config');

const kernel = new Kernel(config);

kernel.boot().catch((err) => {
  console.error('[stratum] Fatal error during boot:', err);
  process.exit(1);
});

Module Contract

Every Stratum module exports an object with a name, version, optional dependencies, and lifecycle methods.

packages/module-example/index.js
module.exports = {
  name: 'module-example',
  version: '1.0.0',
  dependencies: ['module-db-sqlite', 'module-api'],

  // Optional: UI plugin metadata
  ui: {
    routes: [{ path: '/example', component: 'ExamplePage' }],
    nav: [{ label: 'Example', path: '/example', icon: 'Zap' }],
  },

  async register(context) { },   // Declare services & config
  async init(context) { },       // Connect resources, migrations
  async start(context) { },      // Mount routes, go live
  async stop(context) { },       // Graceful shutdown
  async destroy(context) { },    // Cleanup
};
FieldTypeRequiredDescription
namestringYesUnique module identifier
versionstringYesSemver version string
dependenciesstring[]NoModule names this depends on (used for load ordering)
uiobjectNoUI plugin metadata (routes, nav items, slots)
registerasync fnNoDeclare services and configuration schemas
initasync fnNoConnect to databases, run migrations
startasync fnNoMount routes, start listeners
stopasync fnNoStop accepting work, flush buffers
destroyasync fnNoClose connections, free resources

Lifecycle

Modules go through ordered phases. The Kernel runs each phase across all modules (in dependency order) before moving to the next.

Load
→
register()
→
init()
→
start()
→
stop()
→
destroy()
PhasePurposeExample
LoadRequire modules, validate contract, topological sort by dependenciesInternal — handled by ModuleLoader
register()Declare services and register them in the service registryRegister db service, auth.requireAuth middleware
init()Connect to databases, run migrations, prepare stateOpen SQLite connection, create tables
start()Mount HTTP routes, start event listeners, go liveMount Express routers, start HTTP server
stop()Stop accepting new work, flush buffersClose HTTP server
destroy()Release resources, close connectionsClose database connection
Shutdown order On SIGINT/SIGTERM, stop() and destroy() run in reverse dependency order — the last module started is the first to stop.

Context Object

Every lifecycle method receives a context object scoped to the current module.

context = {
  config,      // Module-specific options from stratum.config.js
  services,    // ServiceRegistry instance
  events,      // EventBus instance
  logger,      // Scoped logger: context.logger.info('...')
};
PropertyTypeDescription
configobjectThe options object from this module's entry in stratum.config.js
servicesServiceRegistryRegister and retrieve named services
eventsEventBusSubscribe to and emit events
loggerobjectPrefixed console logger (info, warn, error, debug)

The config namespace is automatically derived from the module name: module module-auth reads config from @stratum/module-auth in stratum.config.js.

Service Registry

The service registry is the primary mechanism for inter-module communication. Modules register named services during the register() phase, and other modules consume them via get().

// Register a service
context.services.register('my.service', implementation);

// Retrieve a service (throws if not found)
const service = context.services.get('my.service');

// Check if a service exists (useful for optional deps)
if (context.services.has('db')) {
  const db = context.services.get('db');
}
No duplicates Registering a service with a name that already exists throws an error. Use has() to check first if needed.

Event Bus

The event bus provides async publish/subscribe for decoupled module communication.

// Subscribe to an event
const unsub = context.events.on('user.created', async (user) => {
  context.logger.info(`New user: ${user.username}`);
});

// Emit an event (handlers run sequentially, awaited)
await context.events.emit('user.created', { id: 1, username: 'admin' });

// Unsubscribe
unsub();
// or: context.events.off('user.created', handler);

Built-in Events

EventEmitted ByPayloadListeners
user.created module-auth { id, username, email } module-rbac (assigns default role)

Configuration

All module configuration lives in stratum.config.js at the project root.

stratum.config.js
module.exports = {
  modules: {
    '@stratum/module-db-sqlite': {
      enabled: true,
      options: { filename: './data/stratum.db' },
    },
    '@stratum/module-api': {
      enabled: true,
      options: { port: 3001, cors: true },
    },
    '@stratum/module-auth': {
      enabled: true,
      options: { jwtSecret: 'change-me-in-production', tokenExpiry: '24h' },
    },
    '@stratum/module-rbac': {
      enabled: true,
      options: { defaultRole: 'user' },
    },
    '@stratum/module-dashboard': {
      enabled: true,
    },
    '@stratum/module-firebase': {
      enabled: false,
      options: {
        credential: './service-account.json',
        // databaseURL: 'https://<project>.firebaseio.com',
        // storageBucket: '<project>.appspot.com',
      },
    },
  },
};
FieldTypeDescription
enabledbooleanSet to false to disable a module without removing code. Defaults to true.
optionsobjectModule-specific settings, accessible via context.config in lifecycle methods.
Dependency enforcement If module A declares a dependency on module B, and module B is disabled, the Kernel will throw an error at boot time. This prevents runtime crashes from missing services.

ui-shell

React + Vite frontend application with dynamic plugin discovery. Not a traditional module — it’s the frontend counterpart to the Kernel.

How It Works

  1. On boot, fetches GET /api/__plugins to discover all module UI metadata
  2. Aggregates routes and nav items into PluginContext
  3. AppRouter dynamically generates React Router routes from plugin metadata
  4. Components are lazy-loaded from the components.js registry
  5. MainLayout renders sidebar nav from plugin nav items

Creating a Module

Step-by-step guide to creating a new Stratum module.

Step 1: Create the Package

mkdir packages/module-my-feature
packages/module-my-feature/package.json
{
  "name": "@stratum/module-my-feature",
  "version": "1.0.0",
  "description": "My custom feature module",
  "main": "index.js",
  "dependencies": {}
}

Step 2: Implement the Module

packages/module-my-feature/index.js
module.exports = {
  name: 'module-my-feature',
  version: '1.0.0',
  dependencies: ['module-api', 'module-auth'],

  ui: {
    routes: [{ path: '/my-feature', component: 'MyFeaturePage' }],
    nav: [{ label: 'My Feature', path: '/my-feature', icon: 'Star' }],
  },

  async register(context) {
    // Expose services for other modules
    context.services.register('my-feature.doSomething', (input) => {
      return `Processed: ${input}`;
    });
    context.logger.info('Registered');
  },

  async init(context) {
    // Set up database tables
    const db = context.services.get('db');
    db.exec(`
      CREATE TABLE IF NOT EXISTS my_items (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        name TEXT NOT NULL,
        created_at DATETIME DEFAULT CURRENT_TIMESTAMP
      )
    `);
    context.logger.info('Tables ready');
  },

  async start(context) {
    const { Router } = require('express');
    const router = Router();
    const registerRouter = context.services.get('api.registerRouter');
    const requireAuth = context.services.get('auth.requireAuth');
    const db = context.services.get('db');

    router.get('/', requireAuth, (req, res) => {
      const items = db.all('SELECT * FROM my_items');
      res.json(items);
    });

    router.post('/', requireAuth, (req, res) => {
      const { name } = req.body;
      const result = db.run(
        'INSERT INTO my_items (name) VALUES (?)', [name]
      );
      res.status(201).json({ id: Number(result.lastInsertRowid), name });
    });

    registerRouter('/api/my-feature', router);
  },

  async stop() {},
  async destroy() {},
};

Step 3: Add to Config

stratum.config.js
'@stratum/module-my-feature': {
  enabled: true,
  options: {
    // your module-specific options here
  },
},

Step 4: Install & Run

npm install
node index.js

The Kernel will automatically discover, sort, and boot your module.

  DashboardPage: lazy(() => import('./pages/DashboardPage')),

  MyPage: lazy(() => import('./pages/MyPage')),  // <-- add this

Creating the Page Component

packages/ui-shell/src/pages/MyPage.jsx
import { useState, useEffect } from 'react';
import { useAuth } from '../AuthContext';

export default function MyPage() {
  const { authFetch } = useAuth();
  const [items, setItems] = useState([]);

  useEffect(() => {
    authFetch('/api/my-feature')
      .then(res => res.json())
      .then(setItems);
  }, [authFetch]);

  return (
    <div className="page">
      <h2>My Feature</h2>
      {items.map(item => (
        <p key={item.id}>{item.name}</p>
      ))}
    </div>
  );
}

Firebase Modes

The Firebase module supports two operational modes depending on which other modules are enabled.

Mode 1: Coexistence

Both SQLite and Firebase are enabled. Firebase services are available under the firebase.* namespace. The db service still points to SQLite.

stratum.config.js
'@stratum/module-db-sqlite': { enabled: true },
'@stratum/module-auth':      { enabled: true },
'@stratum/module-firebase':  { enabled: true, options: { ... } },
ServiceProvider
dbmodule-db-sqlite (SQLite)
auth.requireAuthmodule-auth (JWT)
firebase.firestoremodule-firebase
firebase.authmodule-firebase
firebase.storagemodule-firebase

Mode 2: Firebase-Only

Disable SQLite and JWT auth. Firebase fills the standard service slots so downstream modules (RBAC, dashboard) work unchanged.

stratum.config.js
'@stratum/module-db-sqlite': { enabled: false },
'@stratum/module-auth':      { enabled: false },
'@stratum/module-firebase':  { enabled: true, options: { ... } },
ServiceProvider
dbmodule-firebase (Firestore adapter)
auth.requireAuthmodule-firebase (Firebase Auth)
firebase.firestoremodule-firebase
firebase.authmodule-firebase
firebase.storagemodule-firebase
Dependency note When using Firebase-only mode, you must update module-rbac and module-auth dependencies accordingly, or use the Firebase module as a drop-in by ensuring the dependency chain resolves. The Firebase module only declares module-api as a dependency.

API Reference

ServiceRegistry

// Register a named service (throws if name already taken)
context.services.register(name: string, implementation: any): void

// Retrieve a service (throws if not found)
context.services.get(name: string): any

// Check if a service exists
context.services.has(name: string): boolean

// Get all services as a plain object
context.services.getAll(): Record<string, any>

EventBus

// Subscribe to an event. Returns an unsubscribe function.
context.events.on(event: string, handler: Function): Function

// Unsubscribe a specific handler
context.events.off(event: string, handler: Function): void

// Emit an event. Handlers run sequentially and are awaited.
context.events.emit(event: string, data: any): Promise<void>

// Remove all listeners (used internally during shutdown)
context.events.removeAll(): void

Logger

// All methods prefix output with [module-name]
context.logger.info(...args)   // console.log
context.logger.warn(...args)   // console.warn
context.logger.error(...args)  // console.error
context.logger.debug(...args)  // console.debug

ConfigManager (Internal)

// Check if a module is enabled
configManager.isEnabled(moduleName: string): boolean

// Get module options
configManager.getModuleConfig(moduleName: string): object

// Get all enabled module names
configManager.getEnabledModules(): string[]
